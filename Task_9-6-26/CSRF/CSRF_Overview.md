# Cross-Site Request Forgery (CSRF)

## What is CSRF?

Cross-Site Request Forgery (CSRF) is a web security vulnerability that tricks an authenticated user into performing an unwanted action on a web application without their knowledge or consent.

A CSRF attack exploits the trust that a website places in a user's browser. Since browsers automatically include session cookies with requests, an attacker can cause a victim's browser to send unauthorized requests to a vulnerable application.

### Example

Suppose a user is logged into a banking website. If the user visits a malicious website, the attacker may be able to trick the victim's browser into sending a request that transfers money or changes account settings. Since the browser automatically includes the user's session cookie, the bank may process the request as if it came from the legitimate user.

---

## How Does CSRF Work?

For a CSRF attack to succeed, the following conditions are usually required:

### 1. Relevant Action

The application contains an action that is useful to an attacker, such as:

* Changing an email address
* Changing a password
* Transferring money
* Updating account settings
* Deleting data

### 2. Cookie-Based Session Handling

The application uses session cookies to identify authenticated users.

After login, the server issues a session cookie:

```http
Set-Cookie: session=xyz123
```

The browser stores this cookie and automatically includes it in future requests:

```http
Cookie: session=xyz123
```

### 3. No Unpredictable Parameters

The request does not contain any secret or unpredictable values that the attacker cannot determine.

For example:

```http
POST /email/change

email=attacker@gmail.com
```

Since the attacker knows the URL and parameter values, they can recreate the request.

---

## CSRF Attack Flow

1. The victim logs in to a website.
2. The server creates a session and sends a session cookie.
3. The browser stores the session cookie.
4. The victim visits a malicious website controlled by an attacker.
5. The malicious website causes the victim's browser to send a request to the target application.
6. The browser automatically includes the victim's session cookie.
7. The server receives the request and believes it came from the authenticated user.
8. The requested action is performed.

---

## Why CSRF Works

CSRF works because the browser automatically sends authentication cookies with requests to the target website.

The attacker does not need to know the victim's session cookie. The browser attaches it automatically when sending requests to the target domain.

As a result, the server may be unable to distinguish between a legitimate request and a forged request.

---

## CSRF Protection Mechanisms

### 1. CSRF Tokens

A CSRF token is a random, unpredictable value generated by the server and included in sensitive requests.

Example:

```http
POST /email/change

email=user@example.com
csrf_token=A7X92KLM
```

The server verifies the token before processing the request.

Since attackers cannot predict or obtain the token, forged requests are rejected.

### 2. SameSite Cookies

The SameSite cookie attribute restricts when browsers send cookies with cross-site requests.

Example:

```http
Set-Cookie: session=xyz123; SameSite=Strict
```

This helps prevent browsers from sending session cookies during cross-site requests.

### 3. Origin and Referer Validation

The server can verify the Origin or Referer headers to ensure that requests originate from trusted domains.

If a request comes from an untrusted website, it can be rejected.

### 4. Re-authentication

For sensitive actions, applications can require users to re-enter their password before the action is performed.
This prevents attackers from performing actions through forged requests.







Cross-Site Request Forgery (CSRF) is a vulnerability that allows attackers to perform unauthorized actions on behalf of authenticated users. It occurs because browsers automatically send session cookies with requests. Proper defenses such as CSRF tokens, SameSite cookies, Origin validation, and re-authentication can effectively protect applications from CSRF attacks.
